1System Overview

This is a dual-X gantry. Two parallel rails — X-Left and X-Right — run in the same direction and move together; a Y cross-beam spans between them and travels along their length. Each motor keeps its own FMDD50D40NOM driver, but the two X drivers share one controller axis — they receive the same pulse and direction signals, so the two rails step in perfect hardware lockstep and cannot drift out of sync. The Y beam has its own axis. The gantry is squared mechanically at assembly (it is not squared electronically).

The FMC4030 is a pulse-type, 3-axis motion controller built around a 32-bit ARM chip. It generates pulse (PU) and direction (DR) signals for three external stepper drivers, reads the axis limit/home switches (this build uses four — a positive/negative pair on X-Left and a pair on Y), and provides 4 digital inputs and 4 digital outputs. A host PC connects over Ethernet (default 192.168.0.30 : 8088).

Gantry essentials (read before wiring) Because both X drivers share one controller axis, the two rails are synchronised in hardware — but two things still matter:
  • Wire both X motors identically — no reversal needed. Both rails are mounted the same way (same orientation, motors on the same end), so the shared direction signal moves both carriages the same physical direction. Use the standard coil colours on both motors. (Only a mirror-mounted build would need one motor's A+/A− swapped — not your case.) The §14 jog test confirms direction; see §8.
  • Squaring is mechanical. The controller can't move one rail relative to the other, so the gantry's square comes entirely from the build (§5.1) and is held by the rigid beam.
Build order (recommended) Mechanical assembly → set driver DIP switches → power distribution → motor-to-driver → driver-to-controller → limit switches → communication → pre-power checks → power on & test. Follow the sections below in order.

3Safety & Tools

⚠ Electrical safety The power supply primary side carries 100–220 VAC. All wiring must be done with the system fully de-energized and the AC plug removed. Mains termination should be done by a qualified person. Never work on live terminals.

Do

  • Wire everything with power OFF.
  • Bond the controller PE terminal and the machine frame to protective earth.
  • Double-check polarity (24V+ / 24V−) before powering up.
  • Use ferrules on stranded wire entering screw terminals.
  • Keep motor/driver power wiring separated from signal/Ethernet wiring.

Don't

  • Don't plug/unplug the motor while the driver is powered — it can damage the driver.
  • Don't exceed DC 50 V on the driver power input.
  • Don't connect the motor connector with reversed coil pairs.
  • Don't share the 5 V signal common with the 24 V power ground arbitrarily.

Tools

Small flat & Phillips screwdrivers, Allen/hex key set (for motor & flag mounts), wire stripper/crimper, ferrules, multimeter (continuity & voltage), and a PC with an Ethernet port.

5Mechanical Assembly

Step 5.1 — Set up the gantry frame

Mount the two X rails parallel to each other and the Y cross-beam square (90°) across them. Gantry accuracy depends on this:

  • Set the two X rails truly parallel and at the same height — measure the rail-to-rail distance at both ends and confirm it is equal.
  • Mount the Y beam perpendicular to the rails; check the diagonals of the rectangle are equal to confirm square.
  • Make sure both X carriages start at the same position along the rails before fixing the beam, so the gantry is square at its mechanical start.
Step 5.2 — Mount the motors

Bolt each 57HS motor to its actuator/axis motor mount using four M5 screws (NEMA 23 pattern, 47.14 mm): one on the X-Left rail, one on the X-Right rail, one on the Y beam. Make sure each motor shaft and lead-screw are concentric before tightening the coupling, and tighten the set-screws onto the shaft flats.

Mount both X motors the same way. Install the two X motors in the same orientation, on the same end of their rails (same-hand leadscrews). Then the same shaft rotation moves both carriages the same physical direction, so both motors get identical coil wiring and no reversal is needed (§8). (Only if a build mirror-mounts the two motors would one need its coils swapped — the §14 jog test would reveal it.)
Step 5.3 — Assign & label the axes

Label each motor cable and each driver. Both X drivers are fed from controller axis 1 (the shared X-gantry axis); the Y driver uses axis 3. Axis 2 is unused.

  • X-Left driver/motor → controller axis 1 (5V/PU1/DR1)
  • X-Right driver/motor → controller axis 1 (5V/PU1/DR1) — same terminals, paralleled
  • Y driver/motor → controller axis 3 (5V/PU3/DR3)
Step 5.4 — Install limit-switch flags & sensors

The FSK40 photoelectric sensor mounts to the actuator carriage/rail; the red trigger flag mounts to the moving block so it passes through the sensor slot at each end of travel (see the FSK40 installation photo). With four switches, fit a + and − pair on X-Left and a + and − pair on Y — four total. X-Right gets none.

  • Mount the sensor body so the moving flag enters the optical gap without mechanical contact.
  • Position the positive-end sensor at the far/maximum-travel end, and the negative-end sensor at the home/zero end of each axis.
  • Leave a few millimetres of margin before the hard mechanical stop so the switch trips before the carriage bottoms out.
  • Confirm the indicator LED on the sensor changes state as the flag enters/leaves the slot.
Square the gantry mechanically. The X pair (on LP1/LN1, axis 1) becomes the home/limit for the whole gantry — homing axis 1 moves both motors together, so the gantry slides home as one unit with no fighting. But the controller can't move one rail relative to the other, so squareness comes entirely from the build (Step 5.1) and the rigid beam. Mount the X switch flags on one rail (e.g. X-Left). The opposite rail end has no over-travel switch, so set conservative soft-travel limits in software.
Step 5.5 — Mount controller, drivers & power supply
  • Mount the FMC4030, the three drivers, and the 24 V supply in the control cabinet/panel.
  • If drivers are mounted side-by-side, keep ≥ 50 mm between them for heat dissipation (they have thermal shutdown at 80 °C).
  • Keep AC mains wiring physically separated from low-voltage signal wiring.

6Driver DIP Switch Setup

Set the DIP switches on each FMDD50D40NOM driver before wiring power. There are 8 switches: SW1–SW3 set current, SW4 sets mode, SW5–SW8 set microstepping.

SW4 must be OFF SW4 = OFF selects pulse/subdivision mode — this is required when driven by the FMC4030's PU/DR pulse signals. SW4 = ON puts the driver in I/O mode and the motor will not respond to pulses.

8Motor → Driver Wiring

Each motor connects to its driver's A+/A−/B+/B− coil terminals. The 57HS motor's 4-pin (XH-4) connector colours map directly to the driver:

Motor pinMotor colourCoilDriver terminal
1BlackA (A+)A+
2GreenĀ (A−)A−
3RedB (B+)B+
4BlueB̄ (B−)B−
Coil pairing is what matters. A = Black + Green, B = Red + Blue. If a motor runs rough, vibrates in place, or won't turn, you most likely split a coil pair — verify with a multimeter: the two wires of one coil show continuity (~0.75 Ω), across coils shows open.
Direction. Wire all three motors (X-Left, X-Right, Y) to their drivers using the colour map above. To flip a motor's direction, swap its A+ ↔ A− with power off. For the Y axis you can alternatively invert direction in software — but not for the X rails (they share one axis, so X direction is set only by wiring; see below).
Gantry direction: wire both X motors identically (same-orientation mounting). Both X drivers share one direction signal (axis 1), so both motors always rotate the same way at the same instant. Because both rails are mounted the same way (Step 5.2), that shared rotation moves both carriages the same physical direction — so use the standard coil colours (Black/Green/Red/Blue) on both motors. No A+/A− swap.

Why a swap is sometimes needed (not your case): if the two motors were instead mirror-mounted (one flipped 180°), the mirror would turn the same rotation into opposite travel and rack the gantry; you'd then swap A+ ↔ A− on one motor to cancel it. Since the DIR signal is shared, that correction can only be made in wiring, never in software.

Confirm with the §14 jog test: nudge axis 1 a few mm — both rails should move together in the same direction. If they move apart, one motor is effectively reversed (e.g. an opposite-hand screw or a flipped mount): power off and swap that motor's A+/A−.

9Driver → Controller Signal Wiring

Each driver takes three signals: a 5 V, a pulse (PU), and a direction (DR), in a common-anode connection — the controller's 5 V drives the driver's PU+ and DR+ (tied together), and the controller's pulse/direction pins sink into PU− and DR−. The key point for this gantry: the two X drivers are wired in parallel to the same controller axis (axis 1), so both receive identical pulses and step in lockstep. The Y driver uses axis 3. Axis 2 is unused.

Both X rows reference the same axis-1 terminals — you are landing two wires (one to each X driver) on each of 5V, PU1, DR1. Axis 2 (PU2/DR2) stays empty.

Why this gives perfect sync. With one pulse source feeding both X drivers, the controller physically cannot command the two rails differently — they step together by construction. No software interpolation or "move both axes equally" discipline is needed. The trade-off is that you cannot square the rails electronically (it's mechanical, §5.1) and X direction is fixed in wiring (§8).
Signal load: each driver input draws ~15–17 mA at 5 V; two in parallel (~35 mA) is well within the controller's drive capability. Keep the paralleled signal wires short / run them together. Note the 5 V only feeds the opto-isolator LEDs — it carries no motor current. Each driver makes its own motor current from its 24 V V+/V−.
Optional — split the 5 V across two terminals. The 5V pins on all three axes are the same internal 5 V rail, so if you'd rather distribute the anode load you may take X-Right's PU+/DR+ from the unused Axis-2 5V terminal instead of Axis-1 5V:
X-LEFT PU+/DR+ ← Axis-1 5V PU− ← PU1 DR− ← DR1 X-RIGHT PU+/DR+ ← Axis-2 5V PU− ← PU1 DR− ← DR1 ◄ signals STILL on axis 1
Only the 5 V tap moves — both drivers' PU−/DR− must stay on Axis 1's PU1/DR1, or you lose synchronisation. This is electrically unnecessary (two opto inputs total only ~34 mA, and it doesn't reduce the PU1/DR1 sink current) but harmless. Meter Axis-1 ↔ Axis-2 5V for continuity first to confirm they're the same rail.
Use the controller's 5 V — not 24 V — for PU+/DR+. Bridge axis 1's 5V terminal to both PU+ and DR+ on both X drivers. Do not feed 24 V into the driver's PU/DR inputs in this configuration.
MF (motor-free / enable): Leave MF+/MF− open on all drivers. When MF is energised (active-low) the driver releases the motor (no holding torque). You normally want holding torque, so leave it disconnected.

11Digital I/O & Communication (optional)

11.1 Digital inputs (IN0–IN3)

Four 24 V inputs, active low. Wire a switch/sensor signal to IN0…IN3 and its return to the input GND. Use for external start buttons, sensors, e-stop sense, etc.

11.2 Digital outputs (OP1–OP4)

Four 24 V open-drain outputs (sink ≤ 300 mA). Wire the load between 24 V+ and the output pin; the output pulls the load's low side to ground when active. Use for relays, valves, indicators, etc.

24V+ ──► [ Load / relay coil ] ──► OPx (FMC4030 sinks to GND when ON)

11.3 Host communication — Ethernet (CN3)

  • Connect a standard RJ45 cable from the controller's CN3 to your PC, router, or switch.
  • Controller defaults: IP 192.168.0.30, port 8088.
  • Set the PC's network adapter to the same subnet, e.g. IP 192.168.0.35, mask 255.255.255.0, gateway 192.168.0.1.
  • Verify with ping 192.168.0.30 -t. Replies = link is good.
Other ports (not needed for basic bring-up): CN1 = RS-232 DB9 (pin 2 TX, pin 3 RX) for touch screen/PC; CN2 = RJ45 carrying RS-232 debug and RS-485 (lines 4 & 5). Ethernet is the primary control link.

13Pre-Power Checklist

Walk this list with the system still de-energized and a multimeter in hand.

  • AC mains landed correctly on the supply L / N / earth; cover/strain-relief in place.
  • 24 V bus polarity verified: +24 V from 24V+ to 24V− (no short to PE).
  • Controller VN+/VN− correct; PE bonded to frame and earth.
  • Each driver V+/V− correct polarity.
  • Each motor coil pairs verified by continuity: A = Black+Green, B = Red+Blue.
  • No motor connector to be plugged/unplugged with power on (connect now, while off).
  • Both X drivers paralleled on axis 1: 5V → both PU+ & DR+; PU1 → both PU−; DR1 → both DR−. Y driver on axis 3 (5V/PU3/DR3). Axis 2 (PU2/DR2) empty.
  • Both X motors wired identically (same-orientation mounting → no coil swap); direction confirmed by the §14 jog test.
  • Limit switches (4): Brown→24V+, Blue→24V−, Black→correct input. Gantry X pair → LP1/LN1 + axis-1 GND; Y pair → LP3/LN3 + axis-3 GND. LP2/LN2 unconnected.
  • Gantry squared mechanically at build (it cannot be squared electronically).
  • Driver DIP set on all three: current = 2.0 A (SW1 OFF, SW2 OFF, SW3 ON), SW4 OFF, microstep chosen identically.
  • Ethernet cable seated in CN3; PC NIC set to 192.168.0.x subnet.
  • No loose strands, no terminals touching adjacent terminals, all screws tight.

14First Power-On & Bring-Up

  1. Energize. Apply AC. Confirm the 24 V supply LED is on and the controller/driver power LEDs light. No driver should show a red ALM fault.
  2. Network. From the PC run ping 192.168.0.30 -t and confirm replies.
  3. Connect software. Launch the FUYU controller software, connect to 192.168.0.30 : 8088. (You must interact within ~1 minute or the controller drops idle clients — just reconnect if so.)
  4. Check limit feedback. In the status panel, trip each fitted switch by hand and confirm the matching input toggles (gantry X → LP1/LN1 on axis 1, Y → LP3/LN3 on axis 3). Fix any swapped/active-state issues now.
  5. Verify both X rails move together (do this first, tiny moves). Jog axis 1 a very small amount and watch both rails. They share the signal, so they move together — and with same-orientation mounting they should travel the same physical direction. If the carriages instead move apart (rack/bind), one motor is effectively reversed: power off and swap that motor's A+ ↔ A− (§8), then re-test. Don't make large moves until this is correct.
  6. Jog the gantry (axis 1) and Y (axis 3) at low speed. Confirm:
    • Both X rails move smoothly together, square, no binding (rough motion → recheck coil pairing).
    • Y moves smoothly.
    • Each axis stops when its limit switch triggers.
  7. Home the machine. Home axis 1 to LN1 — both X motors drive together, so the whole gantry slides home as one unit (no fighting). Home Y (axis 3) to LN3. Set this position as zero. Squareness comes from the mechanical build (Step 5.1), not from homing.
For the software phase: the gantry X is a single axis to the controller — in the SDK this is axis index 0 (terminals PU1/DR1); the Y beam is axis index 2 (PU3/DR3). One X command moves both rails in lockstep. No interpolation or "move both axes equally" logic is needed — that's the whole point of sharing the signal.
Hardware bring-up complete. With both X rails moving together squarely, Y jogging, homing working, and all four limits verified, the mechanical/electrical build is done. You're ready to move on to SDK/programming.

16Control Software (FuyuRailController)

With the hardware wired and powered (§1–§14), the rails are driven from a small Qt desktop application — FuyuRailController — that talks to the FMC4030 over Ethernet. It reuses the lab's existing Velmex control panel (the same sliders, Calibrate / Center / Scan buttons and settings dialog) and only swaps the controller underneath to drive the FMC4030 instead of a serial Velmex VXM.

Scope. §1–§15 are assembly & wiring. This section is a short operator guide to the control app; full build / SDK details live in the project's FuyuRailController/README.md.

16.1 Connect

  • Ethernet cable from the controller's CN3 to the PC (§11.3).
  • Set the PC network adapter to the controller's subnet, e.g. 192.168.0.35 / 255.255.255.0.
  • Controller default 192.168.0.30 : 8088; verify with ping 192.168.0.30 -t.
  • Launch the app — a connection dialog appears first, with the controller's IP address and port (a Default button restores 192.168.0.30 : 8088).
  • Click OK — the app tries to connect. On success it opens the control panel; on failure it shows the error and stays so you can correct the address/port. The working values are remembered for next time.

16.2 The control panel

The window stacks one group box per axis: X gantry (SDK axis 0 — both rails) and Y beam (SDK axis 2). Each axis is a 3-row grid of label · slider · value box · button:

┌─ X gantry Controls: ──────────────────────────────────────────┐ │[?] Left Limit: ●─────────────── [ 0.000 mm] [Calibrate]│ │ Slider Position: ────────●─────── [125.000 mm] [Center ]│ │ Right Limit: ───────────────● [250.000 mm] [Scan ]│ └──────────────────────────────────────────────────────────────────┘
ControlWhat it does
Left / Right Limit slidersThe soft travel limits. Normally set automatically by Calibrate; can be nudged by hand.
Slider PositionCommands an absolute move — drag the handle or type a value in the box. It is a setpoint (it jumps, it doesn't scroll).
CalibrateHomes to the negative switch (sets zero), then seeks the positive switch (sets the travel). The slider range then reflects the real travel.
CenterJumps the axis to the middle of its travel.
ScanStepped sweep from left to right in Scan Steps stops (placeholder hook for a camera-capture loop).
? (settings)Units, Home Toward, Speed, Scan Steps — see §16.4.

16.3 First moves

  1. Calibrate each axis first. It homes to the negative/home switch and seeks the positive switch, so the controller learns zero and the full travel from your limit switches (no travel is typed in by hand).
  2. Then move: drag the Position slider, type a value, or press Center. The carriage drives there; the live position streams back.
  3. Scan runs a stepped left→right sweep — wire its per-step signal to your capture routine when integrating with the recorder.
Gantry reminder. The two X rails are one controller axis (shared PU/DR), so "X gantry" moves both rails together — there is no way (and no need) to move one rail alone. Squareness comes from the mechanical build (§5.1).

16.5 Build & run

  • Simulation (default): open FuyuRailController.pro in Qt Creator and Run — a built-in software simulator stands in for the controller, so the whole UI works with no hardware or SDK. Verified with Qt 6.9.3 / MSVC 2022 (64-bit).
  • Real hardware: build with CONFIG+=hardware (Projects ▸ Build ▸ qmake additional arguments). The vendor FMC4030-Dll.dll already sits in FuyuRailController/sdk/ and is auto-copied next to the .exe by the project file — no manual copying.
Runtime-loaded DLL. The SDK library is loaded at run time (via QLibrary), so there is no link-time dependency — a hardware build launches even if the DLL is absent and simply reports disconnected. Use the 64-bit DLL with the 64-bit kit.

16.6 Blocking vs non-blocking SDK

FUYU ships two builds of FMC4030-Dll.dll — blocking and non-blocking — with identical function names but different behaviour. This project uses the non-blocking, 64-bit (X64) build, because the controller is poll-based: it issues one move command and then polls "is the axis stopped?" on a 500 ms timer.

 Non-blocking (used)Blocking (do not use)
Motion call returnsimmediatelyonly when the move finishes
Live position during a moveyes — the poll updates itno — jumps at the end
Stop / abort mid-moveworkscan't until the move ends
Controller thread during a moveresponsivefrozen inside the call
Don't swap in the blocking DLL. With it the app still compiles, connects and completes simple moves — so it can look fine — but you lose live position feedback and the ability to abort a move, a safety concern on a gantry. If you re-extract the DLL, take it from the "non blocking" archive's X64 subfolder (see FuyuRailController/sdk/README.txt).
Under the hood. The app reuses the Velmex LAUVelmexWidget classes unchanged; only LAUVelmexController was reimplemented to drive the FMC4030 (Ethernet, native millimetres, non-blocking moves polled for completion). Internally 1 slider "count" = 1 µm.
